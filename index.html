NeuSym-RAG: Hybrid Neural Symbolic Retrieval with Multiview Structuring for PDF Question Answering

Ruisheng Cao1, Hanchong Zhang1, Tiancheng Huang1, Zhangyi Kang1, Yuxin Zhang1,
Liangtai Sun1, Hanqi Li1, Yuxun Miao1, Shuai Fan2, Lu Chen1, Kai Yu1,
1SJTU X-LANCE Lab, 2AISPEECH,
arXiv Code Data Task Viewer

Abstract

The increasing number of academic papers poses significant challenges for researchers to efficiently acquire key details. While retrieval augmented generation (RAG) shows great promise in large language model (LLM) based automated question answering, previous works often isolate neural and symbolic retrieval despite their complementary strengths. Moreover, conventional single-view chunking neglects the rich structure and layout of PDFs, e.g., sections and tables. In this work, we propose NeuSym-RAG, a hybrid neural symbolic retrieval framework which combines both paradigms in an interactive process. By leveraging multi-view chunking and schema-based parsing, NeuSym-RAG organizes semi-structured PDF content into both the relational database and vectorstore, enabling LLM agents to iteratively gather context until sufficient to generate answers. Experiments on three full PDF-based QA datasets, including a self-annotated one AirQA-Real, show that NeuSym-RAG stably defeats both the vector-based RAG and various structured baselines, highlighting its capacity to unify both retrieval schemes and utilize multiple views.

Framework of NeuSym-RAG

The entire workflow proceeds as follows: - **Parsing**: Firstly, we parse the raw PDF file into a pipeline of functions to segment it in multi-view, extract non-textual elements, and store them in a schema-constrained database (DB). - **Encoding**: Next, we identify those encodable columns in the DB, and utilize embedding models for different modalities to obtain and insert vectors of cell values into the vectorstore (VS). - **Interaction**: Finally, we build an iterative Q&A agent which can predict executable actions to retrieve context from the backend environment (either DB or VS) and answer the input question.

  • Multimodal Vector Encoding
  • Iterative Agent Interaction

The multiview document parsing process transforms raw PDF files into a structured database (DuckDB) through a comprehensive pipeline:

  1. Querying scholar APIs (e.g., arXiv) to obtain paper metadata, enabling metadata-based filtering during retrieval
  2. Using PyMuPDF to split text at different granularities (pages, sections, fixed-length tokens)
  3. Leveraging OCR models (MinerU) to extract tables, figures, and other visual elements
  4. Employing LLMs/VLMs to generate concise summaries of parsed texts, tables, and images

All extracted elements are populated into the symbolic DB. The DB schema is carefully designed in advance to capture the rich structure of PDF documents.

The multimodal vector encoding process transforms structured database content into vector representations:

  1. Labeling columns suitable for vectorization (e.g., long text summaries, image bounding boxes)
  2. Encoding models of both modalities to vectorize text snippets or cropped images
  3. Creating a one-to-one mapping between database cells and vector entries, supplemented with metadata for efficient retrieval

This stage enables semantic search capabilities while maintaining the connection to the structured database.

The iterative agent interaction process enables dynamic interaction with both the database and vectorstore through five parameterized actions:

  • RetrieveFromVectorstore: Dynamic neural retrieval with customizable parameters (query, collection, filters) for semantic search
  • RetrieveFromDatabase: Symbolic retrieval through SQL queries, enabling precise structured data access
  • ViewImage: Extracting and analyzing specific regions of PDF pages using bounding box coordinates
  • CalculateExpr: Mathematical computation for handling numerical questions and reducing hallucination
  • GenerateAnswer: Terminal action that produces the final answer when sufficient context is gathered

The agent follows a thought, action, and observation loop until it can confidently generate an answer.

  • The agent receives a question and relevant context (schemas, PDFs, etc.).
  • It enters a loop of Thought → Action → Observation:
    • Thought: The agent reasons about what to do next.
    • Action: The agent issues a retrieval or answer-generation command.
    • Observation: The environment returns results or feedback.
  • This loop continues, possibly across multiple turns, until the agent is confident enough to output the final answer using GenerateAnswer.
NeuSym-RAG implements instance-specific, execution-based evaluation through 18 carefully designed evaluation functions with optional parameters. These functions are categorized as either subjective or objective, depending on whether they involve judgment from LLMs. For more details about the evaluation framework and available metrics, please visit our evaluation repository.
